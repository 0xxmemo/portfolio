# edit-resume

Edits the resume at `<repo>/resume.html` and (optionally) starts the live preview so the user sees changes as you make them.

## Files in scope

- `resume.html` — the resume itself. Single self-contained HTML file with embedded CSS and a small live-reload SSE client. Gitignored via `resume.**`.
- `bin/preview` — Bun server (`#!/usr/bin/env bun`). Serves `resume.html` on `http://localhost:4173` and pushes a `reload` SSE event whenever the file is written.
- `bin/export` — Bash script. Renders `resume.html` → `resume.pdf` via headless Chrome, overwriting the root PDF. Honors the `@media print` rules.
- `resume.pdf` — gitignored. Originally the user's source resume; now regenerated by `bin/export`. Don't edit by hand.

## Workflow

1. **If the user asked for a preview / wants to see changes live**, check whether the server is already running before starting another one:

   ```bash
   lsof -ti tcp:4173 >/dev/null && echo "already running" || ./bin/preview
   ```

   Start it with `Bash` using `run_in_background: true` so it keeps serving while you work. Tell the user the URL (`http://localhost:4173`). Override the port with `PORT=5000 ./bin/preview` if 4173 is taken.

2. **Make edits with the `Edit` tool**, not `Write` — the file is long and Edit's diff is what the user reviews. Save = browser reloads automatically; no manual refresh.

3. **Preserve the existing structure.** The HTML uses these conventions:
   - `<section>` per major area (Summary / Skills / Experience / Education / Awards), each opened by an `<h2>`.
   - Experience and education entries use `<div class="entry">` with `.entry-head` (org + dates) and `.entry-sub` (role + extra).
   - Awards use `<div class="award">`.
   - Skills are `<ul class="skills">` with `<b>Category:</b> items`.
   - Date format on entries: `YYYY-MM — YYYY-MM` or `YYYY-MM — Present`.

   When adding a new experience entry, copy an existing `.entry` block and adapt it — don't invent new class names.

4. **Don't touch the live-reload script** at the bottom of `resume.html` (the IIFE that opens `EventSource("/__reload")`) or the `.lr-status` styles unless the user asks. They're infrastructure.

5. **Never edit `resume.pdf`.** If the user wants the PDF regenerated from the HTML, suggest printing to PDF from the browser (Cmd+P → Save as PDF) — the stylesheet has a `@media print` block that hides the live-reload chip and tightens margins.

## Common requests

- *"Add a new job at X"* — duplicate an `.entry` inside the Experience `<section>`, in reverse-chronological order (newest first).
- *"Update my title / phone / email"* — edit the `header.identity` block at the top.
- *"Add a skill"* — append to the relevant `<li>` inside `<ul class="skills">`, comma-separated.
- *"Make it shorter / fit one page"* — trim bullet wording rather than removing sections; the `@media print` block already compacts spacing.
- *"Change the colors / font"* — edit the `:root` CSS variables and/or the `body { font-family }` rule.

## Exporting back to PDF

When the user asks to "export", "regenerate the pdf", or "update resume.pdf":

```bash
./bin/export
```

This overwrites `resume.pdf` in the repo root using headless Chrome (auto-discovers Chrome / Brave / Edge / Chromium; override with `CHROME=/path/to/binary`). The script is synchronous and prints the output path + size. The live-reload chip is suppressed in print media, and the SSE client is a no-op when not served over http(s), so export works whether or not `bin/preview` is running.

## Stopping the preview

```bash
lsof -ti tcp:4173 | xargs kill
```

Or use `TaskStop` on the background task id if you started it via `run_in_background`.
